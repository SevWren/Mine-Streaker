## What The Diagnostic Results Actually Say

### The three outcome groups reveal the real structure

| Group | Conditions | n_unknown | What it means |
|---|---|---|---|
| **342 (regression)** | C0, C2, C3, C5 | 342 | Pattern-breaker and Phase2 caps are **completely irrelevant** |
| **468 (worse)** | C1, C4, C6 | 468 | Disabling inter-SA makes things **worse**, not better |

That second row is the most important finding. The hypothesis going into the diagnostic was "inter-SA steals budget, causing the regression." The data says the opposite: **inter-SA is resolving 126 unknowns** (from ~468 to ~342) and should be kept. The regression is not caused by any of the seven tested conditions — it persists identically across all of them.

### Why wider Phase2 caps had zero effect (C3 = C0)

C3 spends **18 seconds more** than C0 and produces **identical results**. If the Phase2 caps were the binding constraint, more time and wider search would have produced some improvement. Instead, Phase2 hits the exact same wall — `n_unknown=342` — and stops. This is the signature of a **hard exclusion**, not a soft parameter limit. Phase2 is not failing to explore deeply enough; it is failing to explore the right region at all.

### The actual root cause: `scan_unknown_cap = 256` is hardcoded

In `repair_phase2.py`, `scan_unknown_cap = 256` is a hardcoded constant — not a `BoardConfig` field. The diagnostic could only toggle config fields. This one was invisible to it.

The hotspot selection function (`_select_hotspot_unknowns`) picks the top-K densest unknown clusters as anchors, then collects all unknowns within radius=6 of those anchors — **capped at 256**. At 342 unknowns, that means **86 cells (25%) are permanently excluded** from driving mine-search. They are never used as anchors and no mine adjacent to them is ever evaluated as a swap candidate.

The geometry of the failure is precise:

| State | Unknowns | Excluded from scan | Exclusion rate |
|---|---|---|---|
| iter10 target | 285 | 29 | 10.2% |
| iter12 wall | 342 | 86 | 25.1% |
| Without inter-SA | 468 | 212 | 45.3% |

The 57 additional unknowns in iter12 vs iter10 push 57 more cells into the excluded set. Some of those cells are almost certainly in resolvable 50/50 clusters. Phase2 literally cannot see them, so no mine adjacent to them is ever proposed, and stagnation is guaranteed.

This also explains why inter-SA reduced unknowns from 468 to 342 but couldn't go further: at 342, the remaining configurations involve mines that can only be found by scanning unknowns beyond position 256 in the sorted list. Inter-SA is constrained by the same scan window.

### The second untested factor: solver `subset_cap=1200` under deadline

During all repair evaluations, `deadline_s` is active, so the solver runs with `subset_cap=1200` instead of 2400. This means every swap candidate is evaluated with a solver that truncates its constraint propagation. A swap that would create a cascade resolution gets scored as "no improvement" because the solver can't see the cascade. This is a separate issue from the scan cap but compounds it: even for unknowns that ARE in the scan window, some beneficial swaps are invisible to the truncated solver.

## What Needs To Happen Next (In Order)

### Step 1 — Fix `scan_unknown_cap` (the primary root cause)

This is a **one-line change** in `repair_phase2.py`. The fix is to make `scan_unknown_cap` dynamic, scaling with the actual number of unknowns:

```python
# Current (hardcoded, causes exclusion at n_unknown > 256)
scan_unknown_cap = 256

# Fix: scale with actual unknown count, add a sensible ceiling
scan_unknown_cap = min(len(unknown_list), 512)
```

Using `min(n_unknown, 512)` rather than just `n_unknown` keeps a ceiling for very large unknown sets where full scanning would be expensive. At 342 unknowns this means all 342 are scanned. At 1,000 unknowns the cap of 512 still provides 50% coverage — better than the current 74.9% at 342 that's already causing failure. The cap should be exposed as a `BoardConfig` knob so future iterations can tune it without source edits.

### Step 2 — Fix solver `subset_cap` under deadline (the secondary factor)

```python
# Current (halves solver completeness during repair)
subset_cap = 2400 if deadline_s is None else 1200

# Fix: same cap regardless of deadline
subset_cap = 2400
```

This makes the repair oracle accurate. The global repair deadline already governs total time — there is no need to also cripple the solver within that deadline.

### Step 3 — Run a targeted verification before any full benchmark

Because the scan_unknown_cap cannot be overridden by the diagnostic script (it's hardcoded), the verification must patch the source. Run the diagnostic with the two-line patch applied, conditions `C0` only, on `250x156` with seeds 300/301/302. If `n_unknown_median` drops below 285, the root cause is confirmed and the fix works. If not, both fixes combined should be tested before concluding the cause is elsewhere.

### Step 4 — Full 3-board × 3-seed benchmark only after Step 3 confirms recovery

The reason to defer the full benchmark is that the full matrix takes ~46 minutes (`2771s / 7 conditions ≈ 395s per condition`, and the full matrix has 3 boards × 3 seeds = 9 runs). Running it before confirming the fix works wastes nearly an hour.

### Step 5 — Structural decisions for the project going forward

Once the `250x156` regression is resolved, there are three remaining open problems in priority order:

**Problem A — `solvable: true` has never been achieved.** The `repair_phase3.py` enumeration fallback handles `n_unknown ≤ 25` but the project consistently lands at 285–342 unknowns, well outside Phase 3's reach. The path to `solvable=true` requires either getting Phase 1+2 repair below the 25-unknown threshold, or redesigning Phase 3 to handle larger unknown sets through probabilistic enumeration (e.g., belief propagation or local constraint solving rather than 2^N exhaustive search). This is the core unsolved problem of the entire project.

**Problem B — The `250x250` runtime gate (180s) has never been met.** The diagnostic shows runtimes of 102–153s on `250x156`. The `250x250` board has ~54% more cells and will be proportionally slower. Before the runtime gate can be met, the Phase 1 budget allocator and Phase 2 stagnation behavior on the larger board need profiling. The solver subset_cap fix (Step 2) may actually help here — fewer wasted solve calls on mis-scored candidates.

**Problem C — The iter13 plan's adaptive Phase2 controller is not needed.** The data shows the regression is not a search-breadth problem — it's a scan-coverage problem. The full APSC with 32 parameters and 4 levels would have been built on top of a broken scanner and would not have helped. It should be shelved entirely unless the scan_cap fix reveals a genuine breadth problem at a different n_unknown threshold.
